# Коришћење System артикала кроз више клијената

## Контекст

Жељени сценарио је да се артикли дефинишу једном, као master подаци System
клијента, а да их више клијената користи у својим раздвојеним процесима.

Пример:

- `System` клијент држи каталог артикала.
- Клијенти `A`, `B`, `C` користе исте артикле.
- Сваки клијент има своје залихе, зоне, локације, налоге и кориснике.
- `System` клијент нема залихе и не ради пријем робе.
- Залихе не смеју да се мешају између клијената.

## Тренутни модел

`ItemData` је `client-assigned` ентитет и мора имати клијента. Артикал без
клијента није предвиђен моделом.

Дељиви артикал зато не треба моделoвати као артикал без клијента, већ као
артикал System клијента.

Ово се уклапа у постојећу логику јер код већ посебно третира артикле System
клијента као потенцијално дељиве master податке.

## Ризик

Главни ризик је picking процес.

Тренутна логика у `PickingStockFinder.findFirstSourceStock(...)` ради овако:

1. Прво тражи залиху клијента излазног налога.
2. Ако нема довољно и артикал је System артикал, тражи залиху System клијента.
3. Ако и даље нема довољно, тражи залиху свих клијената.

То значи да код System артикала постоји ризик да излазни налог клијента `A`
повуче залиху клијента `B`, ако клијент `A` нема довољно своје залихе.

Ово није прихватљиво за сценарио строге client изолације.

## Предложено будуће решење

Ако буде потребна имплементација, предложено је додати две System Properties:

```text
PICKING_USE_SYSTEM_STOCK
PICKING_USE_OTHER_CLIENT_STOCK
```

Ако ове променљиве не постоје, систем треба да се понаша као до сада.

Због компатибилности, подразумевано понашање треба третирати као:

```text
PICKING_USE_SYSTEM_STOCK=true
PICKING_USE_OTHER_CLIENT_STOCK=true
```

## Семантика

Picking увек прво тражи залиху клијента излазног налога:

```text
stock.client = deliveryOrder.client
```

Ако нема довољно залихе:

- ако је `PICKING_USE_SYSTEM_STOCK=true`, сме да тражи залиху System клијента;
- ако је `PICKING_USE_SYSTEM_STOCK=false`, не сме да користи System залиху;
- ако је `PICKING_USE_OTHER_CLIENT_STOCK=true`, сме да тражи залихе других клијената;
- ако је `PICKING_USE_OTHER_CLIENT_STOCK=false`, не сме да користи залихе других клијената.

## Комбинације

```text
Обе променљиве не постоје
=> старо понашање
```

```text
PICKING_USE_SYSTEM_STOCK=false
PICKING_USE_OTHER_CLIENT_STOCK=false
=> користи само залиху клијента излазног налога
```

```text
PICKING_USE_SYSTEM_STOCK=true
PICKING_USE_OTHER_CLIENT_STOCK=false
=> користи залиху клијента излазног налога и System залиху,
   али никада залиху другог реалног клијента
```

```text
PICKING_USE_SYSTEM_STOCK=false
PICKING_USE_OTHER_CLIENT_STOCK=true
=> користи залиху клијента излазног налога и залиху других клијената,
   али не користи System залиху
```

Последња комбинација је технички могућа, али пословно необична.

## Препоручена вредност за строгу изолацију

За сценарио где `System` држи само артикле, а свака роба припада конкретном
клијенту:

```text
PICKING_USE_SYSTEM_STOCK=false
PICKING_USE_OTHER_CLIENT_STOCK=false
```

Тако се омогућава један заједнички каталог артикала, али се спречава
међуклијентско трошење залиха.

## Место будуће имплементације

Најчистије место за промену је:

```text
PickingStockFinder.findFirstSourceStock(...)
```

Постојеће понашање:

```text
клијент налога -> System клијент -> сви клијенти
```

Предложено понашање:

```text
1. Увек пробај клијента налога.
2. Ако нема довољно и PICKING_USE_SYSTEM_STOCK=true, пробај System клијента.
3. Ако нема довољно и PICKING_USE_OTHER_CLIENT_STOCK=true, пробај остале клијенте.
4. Ако није дозвољен fallback, врати да нема довољно залихе.
```
